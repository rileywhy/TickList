# TickList — Code Review, 2026-07-18

Branch `classes/data-preservation` @ `a23a40d`.

This review is scoped to what the previous audit (`roadmap.md`, written at `2aea1da`) could **not**
have seen: the grade-mapping subsystem that landed afterwards, plus a cross-file pass over the
auth/import/UI contracts. It also re-triages every finding in `roadmap.md` Part 1 against current

## Critical

### C1 — `POST /ticks` is a mass-assignment IDOR: any user can overwrite and steal any other user's tick

`TickController.java:41-44`

```java
@PostMapping({"/ticks", "/tick"})
public Tick createTick(@RequestBody Tick tick, @AuthenticationPrincipal User user) {
    gradeMappingService.applyGradeMapping(tick);
    tick.setUser(user);
    return tickRepository.save(tick);
}
```

The request body binds straight onto the `Tick` **entity** and `id` is never cleared.
`JpaRepository.save()` on a non-null id performs a `merge`, not an `insert`.

Repro: Bob, a legitimately logged-in user, POSTs `{"id": 42, "climbName": "pwned"}` where 42 is
Alice's tick. Spring merges over row 42; `setUser(user)` then reassigns ownership to Bob. Alice's
tick vanishes from her list, and every field Bob didn't send (`notes`, `location`, `stars`,
`pitches`, `climbHeight`, `personalGrade`) is nulled by the merge.

`GET`/`PUT`/`DELETE` are all correctly guarded by `findByIdAndUser`. `POST` is the one unguarded
write path. The frontend never sends `id`, which is exactly why it went unnoticed.

`TickOwnershipIntegrationTest` has 10 tests including "POST ignores a `user` in the body" — but
never "POST with someone else's `id`". The suite reads as complete while leaving the hole open.

**Fix:** `tick.setId(null)` in `createTick`, or bind a `TickRequest` DTO instead of the entity
(preferred — see M4). Add the missing test.

### C2 — An expired token puts the app in a permanently broken state that never prompts re-login

`SecurityConfig.java:26-42`, `JwtAuthenticationFilter.java:61-63`, and every frontend caller.

`SecurityConfig` configures no `authenticationEntryPoint`, so Spring Security 6 falls back to
`Http403ForbiddenEntryPoint`. `JwtAuthenticationFilter` catches `JwtException`, clears the context,
and calls `filterChain.doFilter` anyway — so an expired token reaches `anyRequest().authenticated()`
as *anonymous* and yields **403, never 401**.

Every frontend caller has this shape:

```ts
if (response.status === 401) { onAuthExpired(); ... }
if (response.status === 403) { setMessage("The backend rejected this request (403)."); }
```

So after `security.jwt.expiration-ms` (1 hour) the user is still rendered as logged in —
`AccountMenu` shows their name, `App.tsx` still routes to `/ticks` because `currentUser !== null` —
while every load, create, edit, delete and import fails with "The backend rejected this request
(403)" forever. `onAuthExpired` is dead code. There is no refresh path. The user must manually
navigate to `/login`.

That a 403 branch already exists in all four components suggests someone hit this and papered over
the symptom rather than the cause.

**Fix:** add an `authenticationEntryPoint` returning 401, and have the filter distinguish
"no token" from "bad token". Then pin it with a test (see H3).

## High

### H1 — `tick.gradeValue` silently mixes two incompatible numeric scales

`GradeMappingService.java:45-47` stores `mapping.getSystemOrder()` when a mapping resolves and falls
back to `parsedGrade.gradeValue()` when it doesn't. **These are different scales.** Comparing every
seed row's `systemOrder` against what `GradeParser` computes for the same string: **67 rows disagree.**

| grade | seed `systemOrder` | parser value |
|---|---|---|
| `5.10` | 10.375 | 10.0 |
| `5.8-` | 7.7 | 8.0 |
| `4B` (Font) | 4.0 | 4.333 |
| `5C+` (Font) | 5.583 | 5.833 |
| `AI3` | 2.6 | 3.0 |
| `A6` | 5.25 | 6.0 |

The divergence is deliberate in the seed (sub-6A Font letters are *synonyms* folded onto numeric
rungs; `AI n` sits 0.4 below `WI n`), but `GradeParser` does naive letter-thirds arithmetic and
treats `WI`/`AI`/`M`/`A`/`C` identically. Two ticks both graded Font `4B` — one mapped, one not —
sort 0.33 grades apart in the same column. This directly undermines the leaderboard/comparison
feature the subsystem exists to enable.

**Fix:** the fallback should be `null`, not a value on a foreign scale. Nothing in the suite asserts
`getGradeValue()` at all, which is why this is invisible.

### H2 — Gym ticks in ICE / MIXED / AID / E_Grade can never resolve a mapping

`GradeMappingService.java:77-79` short-circuits: any tick with `discipline == GYM` looks up
discipline `GYM` regardless of system. But the CSV only has GYM rows for `YDS`, `FRENCH_SPORT`,
`FONT`, `V_SCALE` (78/84/73/72 rows respectively; **zero** for the other four).

A gym dry-tooling tick (`discipline=GYM`, `grade="M5"`) parses as `MIXED_M`, looks up
`(MIXED_M, GYM, "M5")`, finds nothing, and silently gets a null `difficultyScore` plus a
foreign-scale `gradeValue` per H1. `GradeSeedCoverageIntegrationTest` can't catch it — its
enumeration only pairs `GYM` with the four systems that do have rows.

### H3 — The auth tests are written to pass under either behaviour, which is what hid C2

`TickOwnershipIntegrationTest.java:192,229`

```java
mockMvc.perform(get("/ticks")).andExpect(status().is4xxClientError());
```

`is4xxClientError()` is satisfied by both 401 and 403. The one test that could have pinned the
contract the frontend depends on deliberately doesn't. Fixing C2 and later regressing it would both
stay green. Same for `importRequiresAuthentication`.

### H4 — Editing a grade without also changing the system dropdown corrupts the mapping

`TickController.java:65` copies the client's `gradeSystem`, and `GradeMappingService.resolveGradeSystem`
(`:69-70`) treats any non-`UNKNOWN` requested value as authoritative. The edit form carries the old
`gradeSystem` forward (`TickCard.tsx:242`).

So editing a grade from `5.11a` to `V4` without touching the dropdown leaves `gradeSystem=YDS` on a
V-scale grade: the mapping lookup fails and `gradeValue` falls back to the YDS parse of "V4" (null).
`TickUpdateProvenanceIntegrationTest` only covers same-system edits.

## Medium

- **M1 — Import is non-transactional, non-idempotent, and its errors are unreadable.**
  `Importer.importCSV` has no `@Transactional` and saves row-by-row, so a malformed cell on row 500
  throws after rows 1–499 have committed. Retrying then duplicates them: `externalId` is never
  populated even though the MP `URL` column carries the route id, and there's no dedupe query.
  Meanwhile `ImportController` puts the message in a `ResponseStatusException` reason, but
  `server.error.include-message` defaults to `never` and `UploadPage` doesn't read the body anyway —
  the user just sees `Could not import CSV (400)`. `skippedRows` is incremented nowhere, so the UI
  always says "Skipped 0", and `ImportBatch` is written by no code path.

- **M2 — `GET /ticks` serializes the full nested `User` on every tick.** `Tick.user` has no
  `@JsonIgnore` (contrast `gradeMapping`, which has one), so every response echoes the owner's email
  and full name. It's the caller's own data today, so this is self-disclosure rather than a leak —
  but it becomes a real leak the moment any shared or cross-user view ships, which is Phase 5.3.

- **M3 — The seeder is upsert-only, non-transactional, and resurrects deactivated rows.**
  `GradeMappingSeeder.java:53` unconditionally re-asserts `setActive(true)` every boot, so a manual
  deactivation is undone on restart, and a row dropped from a regenerated CSV is never deactivated —
  it keeps serving the old score. 817 individual `findBy` selects with no `@Transactional` means a
  mid-batch failure leaves a partially retuned table. One malformed row bricks startup. There is no
  seeder test of any kind.

- **M4 — Entities are still bound directly as request bodies, with no `@Valid` anywhere.** This is
  the root cause of C1, and of registration 500ing on a missing password or duplicate email
  (`UserController.java:60-64` — no `@Valid`, no try/catch, no `@RestControllerAdvice` in the
  codebase). Constraint violations surface as 500s rather than 400/409.

- **M5 — The generator's coverage check validates strings, not the Java lookup path.**
  `docs/gen_grade_mappings.py:757-778` tests whether an MP `Rating` string appears in
  `known_raw_grades`, ignoring `gradeSystem`, `discipline`, case, and the Java normalizer. It
  therefore prints green for ~150 rows `GradeParser` can never classify (`6A/6A+`, `6A/+`, numeric
  Font `4+`, plain French `1`/`3+`, `WI3-4`, `AI2-3`, `M5-6`, bare `E5`, `E1/2`, `HVS`, `3rd`,
  `Easy 5th`). The seed data itself is clean — monotonicity and duplicate-key checks pass with zero
  violations.

- **M6 — Parser drops the modifier on letterless YDS.** `GradeParser.java:179-181` discards `-` and
  honours only `+`, so `5.8-` and `5.8` both return `8.0` while the seed has them at 7.7 and 8.0.
  The `5.9-` / `5.8+` pair inverts relative to the seed.

- **M7 — `RegisterForm.tsx` posts a field the backend has no concept of.** It sends
  `{username, email, password}`; `User` has `firstName`/`lastName` and no `username`. It also sends
  an `Authorization` header to a `permitAll` endpoint. Currently unreferenced (`RegisterPage` has
  its own correct inline form), so it's a landmine rather than a live break.

- **M8 — Six persisted fields remain invisible in the UI.** `pitches`, `stars`, `userStars`,
  `personalGrade`, `climbHeight`, `rawGrade` are populated by the importer and appear in neither
  `TickPayload`, `TickFormValues`, nor `TickCard`'s details — including the "Show Details" modal
  that claims to be the complete view. They now survive edits (per `a23a40d`) but are write-only
  from the user's perspective.

- **M9 — Session is memory-only.** `App.tsx:12` holds `currentUser` in `useState` with no
  persistence and there's no `/me` endpoint, so any refresh silently logs the user out while the JWT
  is still valid. This partially *masks* C2: users who habitually refresh re-login before expiry.

## Low

- `GradeMappingService.java:104` — `case UNKNOWN -> compactGrade` is unreachable; line 51 already
  returned empty for `UNKNOWN`.
- `normalizeEGrade` (`:120-130`) silently drops any token past the second.
- `CorsConfig` is entirely inert in dev — all fetches are relative and go through the Vite proxy,
  which additionally strips the `Origin` header. It will need real values the moment the frontend is
  deployed to its own origin, and nothing tests it.
- `GET /users` is a live method behind `denyAll()` that would dump every user record if that line
  were ever reordered.
- `Grade.java` is dead code — zero references. Delete it.
- `ticklist-issues.spec.ts`'s header claims every test documents an unfixed defect, but two are now
  passing regression guards. A reader following the comment would re-add `test.fail()` and break them.

## Re-triage of `roadmap.md` Part 1

Verified against current code, not against the roadmap's own (drifted) line numbers.

**12 FIXED · 8 PARTIAL · 14 STILL BROKEN · 1 audit error**

Fixed and well-covered: per-user authorization (#1), import ownership (#2), the edit-wipes-data
double bug (#6), the non-array `/ticks` crash (#8), backend tests in CI (#9), ownership tests (#10),
Font-vs-French by discipline (#33), the grade axis itself (#32), `docs/` tracked (#29), backend
integration tests (#35).

Still broken and worth queueing: re-import duplication (#3), import transactionality (#4),
registration 500s (#7), `V0-1 PG13`-style protection ratings (#11), boulder `Attempt` → `SEND` (#12),
the MP `-1` star sentinel (#14), `skippedRows` (#15), BOM/multipart/`ImportBatch` (#16),
`ddl-auto=update` with no Flyway (#19), no `Climb` entity (#20), token persistence (#21),
login/delete error handling (#22), enum-code search (#23), `sourceUrl` XSS (#25), CORS (#31).

Two corrections to the roadmap itself:

- **#13 is an audit error, not a regression.** `DateParser.java:47` uses
  `OffsetDateTime.parse(...).toLocalDate()`, which yields the date *in the original offset* — it
  never converts to UTC, so there is no day shift. Time-of-day is still discarded, but that's a
  feature gap, not the bug described. Phase 2 shouldn't "fix" working code.
- **#6's fix is by omission, not by copying.** `updateTick` preserves the imported fields because it
  loads the managed entity and never touches those setters. Correct and test-covered, but *implicit*:
  the invariant is "the setter list at `TickController.java:59-73` must stay exactly the frontend's
  editable set." Adding a field to `toTickPayload` without a matching setter silently no-ops; adding
  a setter for a non-form field re-opens the original wipe bug. A DTO makes this structural.

1. **C1** — one line (`setId(null)`) plus the missing test. Smallest fix, largest blast radius.
2. **C2 + H3** — 401 entry point, and tighten `is4xxClientError()` to `isUnauthorized()`.
