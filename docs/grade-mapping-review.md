# Grade Mapping — Branch Review & Roadmap

Review of the `grade-mapping` branch (commit `6d4b741`) against `main`, focused on
(a) bugs in the current commit and (b) design holes that get expensive to fix later,
judged against the product goals: cross-discipline difficulty comparison, cross-user
taste/recommendation matching, climb-suitability scoring, clean automatic imports from
any source, and leaderboards.

Produced from a multi-agent review (6 reviewers + adversarial verification). This doc
complements [ROADMAP.md](ROADMAP.md) (the broader codebase audit) and
[grade-axis.md](grade-axis.md) (the axis design); it does not repeat pre-existing
issues from ROADMAP.md unless this branch changed their shape.

---

## What the branch adds

A universal difficulty axis. `GradeMapping` (now a live entity with getters/setters) is
seeded on every boot by `GradeMappingSeeder` from `grade-mappings.csv` (816 rows,
generated by `docs/gen_grade_mappings.py`). `GradeMappingService.applyGradeMapping(tick)`
normalizes a tick's grade, resolves its `GradeSystem`, looks up a `(system, discipline,
rawGrade)` row, and stamps `tick.gradeMapping` (FK) + `tick.gradeValue`. It runs on
import and on tick create/update. `Tick.getDifficultyScore()` is a `@Transient`
delegate; the frontend shows a "Difficulty" row. The branch also adds real backend
integration tests (`TickOwnershipIntegrationTest`) and an H2 test profile.

The direction is right and the seed data is impressively considered. The gaps below are
mostly about *how grades reach the table*, *what happens to the table over time*, and
*what the axis exposes to the features that will consume it*.

---

## Fixed in this pass

Two low-risk fixes applied directly, both inside this branch's new `GradeMappingService`,
plus a new `GradeMappingServiceTest` (the service had zero tests before):

1. **Split-brain lookup** — `applyGradeMapping` parsed the `GradeSystem` from `grade`
   (grade-first) but looked the mapping up from `rawGrade` (rawGrade-first). When the two
   differ (any client that round-trips `GET /ticks` JSON back through `PUT` after editing
   the grade), the edited grade was displayed while the *old* grade's `difficultyScore`
   and `gradeValue` persisted. Now both the parse and the lookup use the same cleaned
   grade string. (`GradeMappingService.java:33`)

2. **`V4/5` slash range** — normalized to `V4/5`, which the seed doesn't contain (it uses
   the dash form `V4-5`), so slash-typed boulder ranges silently got no difficulty. Now
   normalized to the dash form. (`GradeMappingService.java:103`)

3. **Font vs French decided by letter case** (was the first High issue below) —
   `GradeParser` now takes a `Discipline` hint: `parse(String, Discipline)` /
   `parseGradeSystem(String, Discipline)`. When the case-based parse lands on the
   Font/French ambiguous pair, `BOULDER → FONT` and `SPORT`/`TRAD → FRENCH_SPORT`;
   `GYM`/`UNKNOWN`/null keep the case-based guess. `applyGradeMapping` passes
   `tick.getDiscipline()`, and the Mountain Project importer resolves the discipline
   from the CSV's Route Type column *before* the grade parse and feeds it in (it
   previously pre-set `gradeSystem` from a case-only parse, which `resolveGradeSystem`
   then trusted). Covered by new tests in `GradeParserTest`, `GradeMappingServiceTest`,
   and `ImporterTest`. (`GradeParser.java:29`, `GradeMappingService.java:24`, `Importer.java:77`)

4. **Sub-6A Font letter grades had no seed rows** (was the first Medium issue below) —
   the generator now emits letter synonym rows for `FONT,BOULDER` and `FONT,GYM`
   (`3A`–`5C+`): letters fold onto the numeric rungs as the low/mid/high thirds of the
   number (`4A=4-`, `4B=4`, `4C=4+`, `5A=5-`, `5B=5`, `5C=5+`; `3B`/`3C` interpolate
   thirds toward `4-` since no numeric `3±` rungs exist), and plus forms sit at the
   midpoint of adjacent letter rungs, so `5C+` lands halfway between `5+` and `6A`. It
   also fills the French letter gaps the parser accepts but the anchor table didn't
   name: `3a`–`3c+` (thirds between `3` and `4a`) and `4a+`–`4c+` (midpoints — `4c+`
   computes to exactly the existing plain `4+`, which is the fold identity). 762 → 816
   rows; the generator's monotonicity/duplicate/anchor validation passes. The whole
   *class* of bug is now guarded by `GradeSeedCoverageIntegrationTest`, which feeds
   every parser-recognizable real-world grade string (~800 grade × discipline × case
   combinations) through the real `applyGradeMapping` path against the seeded H2 table
   and lists every miss — verified red against the pre-fix CSV (90 distinct misses) and
   green after regeneration. A parser change that outruns the seed now fails the build
   for any form in the test's enumeration; the test documents the contract that a parser
   change which recognizes a *new* form must add it to the enumeration, which then forces
   a seed row.
   (`docs/gen_grade_mappings.py`, `grade-mappings.csv`, `GradeSeedCoverageIntegrationTest.java`)

Everything below is left for planning, not fixed, because each needs a decision or a
migration, not a one-liner.

## Part 1 — Issues in the current commit

### High — correctness or data-integrity, worth addressing before this feature is trusted

- ~~**Font vs French decided by letter case picks the wrong difficulty ladder.**~~
  **Fixed** — see "Fixed in this pass" #3. One follow-up it exposed is listed under
  Medium (sub-6A Font letter grades).

- **No unique constraint on `(gradeSystem, discipline, rawGrade)` + find-or-create seeding
  on every boot.** Two instances booting against one Postgres (rolling deploy,
  blue-green, two devs on a shared DB — the multi-instance shape the leaderboard goal
  needs) can both find no rows and both insert, doubling every key. After that,
  `findBy…AndActiveTrue` throws `IncorrectResultSizeDataAccessException` on the hot path,
  so **every** tick create/update/import row 500s, and the seeder's own `findBy` throws on
  the next boot — permanent outage until someone hand-writes dedup SQL.
  **Partially fixed:** the `@UniqueConstraint`
  (`uk_grade_mapping_system_discipline_raw_grade`) is now on the entity, so the database
  rejects duplicates from any writer — the silent-corruption outcome is gone. Two
  follow-ups deliberately left lingering:
  - **The seeder doesn't tolerate losing the race.** With the constraint in place, the
    losing instance's `saveAll` now throws `ConstraintViolationException` out of the
    `ApplicationRunner` and that instance's *boot fails* — correct data, crashed replica.
    Fix shape: catch the violation and re-read (the row that "wasn't there" now is —
    switch from insert to update), or a native `INSERT … ON CONFLICT DO UPDATE` upsert.
    Note the single end-of-pass `saveAll` makes tolerance awkward: one lost row fails
    the whole 816-row batch in one transaction, so move to per-row/chunked saves first.
    (`GradeMappingSeeder.java:53`)
  - **`ddl-auto=update` may silently skip the `ALTER` on an existing table.** Hibernate's
    `update` mode is unreliable about adding unique constraints to tables that already
    exist, so any dev/prod Postgres whose `grade_mapping` table predates the annotation
    may not actually have the constraint — verify with `\d grade_mapping`, or drop the
    table once and let the seeder rebuild it. The durable fix is real migrations
    (Flyway/Liquibase), which the backfill/re-map job will want anyway.
    (`application.properties`)

- **No backfill — existing ticks are stranded at `difficultyScore = null` forever.**
  `applyGradeMapping` runs only on write paths. Nothing re-resolves existing ticks: not
  the seeder, not a migration, not a read path. So (a) every tick created on `main` before
  this branch, and (b) every tick whose grade was missing from the CSV at write time, stays
  unmapped permanently. Concretely: user A imported 500 MP ticks last month, user B imports
  the identical CSV tomorrow — on a leaderboard summing `difficultyScore`, A scores ~0 and
  B scores full value for the *same climbs*. Every legacy card shows "Difficulty: Unknown".
  Cheap now (single-user era): after seeding, run `applyGradeMapping` over
  `findByGradeMappingIsNull()` — idempotent and small. Brutal later (it also mutates
  `grade`/`rawGrade`, so a years-later sweep needs auditing). (`GradeMappingSeeder.java`)

- **Seeder is upsert-only: rows removed or renamed in a regenerated CSV live forever as
  `active = true`.** `grade-axis.md`'s stated retuning workflow is "edit the generator,
  regenerate, redeploy." But if a regeneration drops or renames a `rawGrade` form, the old
  DB row survives, keeps matching `findBy…AndActiveTrue`, and serves a stale score that
  now disagrees with the rest of the axis — silently. `active` is written but nothing ever
  sets it false. The seeder should deactivate DB rows absent from the CSV. (`GradeMappingSeeder.java:53`)

### Medium — mapping coverage and consistency

- ~~**Sub-6A Font letter grades (`4a`–`5c+`) now resolve to FONT but have no seed rows.**~~
  **Fixed** — see "Fixed in this pass" #4 (letter synonym rows for Font and French,
  guarded by the new `GradeSeedCoverageIntegrationTest`).

- **Declared `gradeSystem` is trusted blindly.** `POST /ticks {grade:"7b",
  gradeSystem:"FRENCH_SPORT", discipline:"BOULDER"}` for a Font 7B boulder returns 200 and
  silently binds a sport mapping ~20 axis points low, looking fully resolved. When the
  declared and parsed systems disagree, either reject (400 naming both) or degrade
  confidence — never store a parser value computed under one system on a tick labeled
  another. (`GradeMappingService.java:63`)

- **Grade forms that the seed data prices but the parser can't classify → silent misses.**
  On the real `inputs/ticks.csv`, protection-suffixed boulders (`V0 R`, `V1 PG13`) and
  range grades (`WI3-4`, `AI2-3`, `M5-6`, `E2/3`) parse as `UNKNOWN` and get no difficulty,
  even though exact rows (`V0`, `WI3-4`…) sit unused in the CSV. The V-scale classifier
  needs the trailing-suffix tolerance YDS already has (`(?:\s.*)?`); the WI/AI/M/E
  classifiers need range/slash forms. (`GradeParser.java:7,11`)

- **`Easy 5th` / `3rd` / `4th` can never match** — `normalizeRawGrade`'s `firstToken`
  truncates `Easy 5th` to `easy`, so even a client declaring `gradeSystem = YDS` misses the
  seeded row. (Also blocked upstream: `GradeParser`'s YDS regex requires `5.x`, a known
  follow-up.) (`GradeMappingService.java:96`)

- **`tick.gradeValue` is a write-time *copy* of `systemOrder`.** `difficultyScore` lives
  behind the FK and picks up retunes, but `gradeValue` is snapshotted onto the tick row and
  doesn't. After a retune, two ticks of the same grade sort as different difficulties
  depending on when they were written. Either make `gradeValue` a transient delegate too,
  or re-stamp it in the backfill job. `grade-axis.md`'s claim that "every existing tick
  picks up the new placement" is only true for `difficultyScore`, not `gradeValue`.
  (`GradeMappingService.java:40`)

- **`PUT` now fabricates `rawGrade` provenance instead of dropping it.** `TickPayload` omits
  `rawGrade`, so on edit it arrives null and `applyGradeMapping` backfills it from the
  cleaned display grade. An imported `5.10a/b` becomes `5.10a` and `rawGrade` now *claims*
  the import said `5.10a` — which breaks any future re-import dedup keyed on raw strings.
  This is the roadmap's known "UI edit wipes data" bug (#6) changed in shape from silent
  null to silent overwrite. Don't overwrite `rawGrade` on update when the body's is null.
  (`TickController.java:65`)

- **Misses are completely silent.** No unmapped counter in `ImportResult`, no log, no
  coverage query. Combined with no re-map path, every parser/normalization fix strands the
  already-imported ticks it would have helped. Add an unmapped count to imports and a batch
  re-map job. (`GradeMappingService.java:39`)

### Low

- **Fallback `gradeValue` uses the parsed system's scale even when the declared system
  differs** — declare `V_SCALE`, type `6a` (parses French, value 6.0), lookup misses, and
  the tick sorts as a V6 among the user's boulders. Only fall back when parsed system ==
  resolved system. (`GradeMappingService.java:42`)
- **`VS 4c` / `HS 4b` (UK adjectival + tech) normalize correctly but have no CSV rows**,
  while bare `VS` and `E1 5a` do — future UK imports miss the most common sub-E1 forms.
  (`GradeMappingService.java:123`)
- **One malformed CSV row bricks boot** — `GradeSystem.valueOf` / `Double.valueOf` /
  missing header throw straight out of the `ApplicationRunner`, taking the whole app (and
  every integration test) down. Adding `UIAA` to the generator per the doc, then deploying,
  would do exactly this. Per-row try/catch; fail only on empty/bad-header. (`GradeMappingSeeder.java:38`)
- **Gym `3rd`/`4th`/`Easy 5th` score `19.0` vs outdoor `12.5`** — the gym curve is clamped
  to a floor, so at the very bottom of the scale (scramble grades) gym reads *harder* than
  outdoor, inverting the "gym is soft low" guarantee. Harmless in practice (nobody
  leaderboards a 4th-class scramble) but a real data artifact; the doc's crossover is also
  stated as score 85 while the data crosses just above V10 (gym V10 = 84). Cosmetic.
  (`grade-mappings.csv`)

## Part 2 — Roadmap: what's next

Ordered so the cheap-now / brutal-later items come first. These are all things the
single-user era makes painless and the multi-user era makes migrations.

### Now (small, unblocks trust in the feature)

1. **Backfill + re-map job.** One idempotent runner that re-resolves ticks with
   `gradeMapping = null` (and re-stamps `gradeValue`). This single mechanism closes four
   findings at once: legacy ticks, mid-history CSV additions, post-parser-fix misses, and
   retune drift. Build it before the data grows.
2. **~~Unique constraint~~ + tolerant seeder + deactivation of removed rows.** The unique
   constraint is done (see the High item above, incl. its two lingering follow-ups); the
   race-tolerant seeder and deactivation of CSV-removed rows remain. Together they make
   the boot seed safe under multiple instances and make the documented retuning workflow
   actually able to remove/rename a grade.
3. **Expose `confidence` (and ideally `systemOrder`) alongside `difficultyScore`.** The
   seed encodes a huge quality spread (YDS 1.0, gym/E-grade 0.4–0.6, WI 0.2–0.3) that no
   consumer can see. A leaderboard `ORDER BY difficultyScore` today ranks a 0.4-confidence
   modeled gym V13 above a real 5.14d. Add a small `{score, confidence}` shape now so the
   first leaderboard/rec code is built with the hedge from day one. This pairs with the
   move away from serializing the JPA entity (ROADMAP Phase 0's DTO work).
4. **Grade-string → mapping resolution hardening.** Font/French disambiguation via
   `discipline`, declared-system mismatch handling, V-scale protection suffixes, WI/AI/M/E
   range forms. Best done as the `parse(String, Discipline hint)` refactor (ROADMAP Phase
   4) rather than piecemeal regex edits — it's the same root cause. When a parser gap
   closes, add the newly classifiable forms to `GradeSeedCoverageIntegrationTest`'s
   enumeration so they're pinned to seed rows from day one.
5. **Make the axis queryable.** `difficultyScore` is `@Transient`, so it can't appear in a
   SQL/JPQL `ORDER BY` and the natural join is an INNER join that would silently drop aid
   and unmapped ticks. Leaderboards and pyramids need to sort/aggregate on it — decide now
   whether that's a persisted denormalized column (kept in sync by the re-map job) or a
   join with an explicit `LEFT JOIN` + null policy.

### Before the next importer (Kaya — ROADMAP Phase 2)

6. **Introduce the `indoor` flag and route the GYM ladder off it, not off
   `Discipline.GYM`.** The branch's own docs say indoor ticks keep `discipline =
   BOULDER/SPORT` and look up the GYM ladder via the indoor flag — but the code keys on
   `Discipline.GYM`. As written, every Kaya gym V4 will score `65.0` (outdoor) instead of
   `56.25` (gym), inflating every gym climber 1–2 grades on cross-user leaderboards and
   poisoning taste/suitability training data. This is one boolean now; a bespoke backfill
   over all ticks later. Do it before Kaya, not after.
7. **Per-source parsers + idempotent imports** (ROADMAP Phase 1) — the grade axis makes the
   "imports from any source, clean and automatic" goal reachable, but only once each source
   has its own parser and a deterministic `externalId` for dedup. Re-importing today
   duplicates; that has to be solved before "all automatic" is safe.

### For the recommendation / taste-matching goals (ROADMAP Phases 3, 5–6)

8. **A `Climb` entity is the real keystone.** Grade mapping currently lives on `Tick`;
   taste matching and "does this climb suit me" need per-*climb* difficulty, consensus
   stars, and style tags, with a cross-source join key so two users ticking the same boulder
   share it. When `Climb` lands, the axis lookup should move to (or be shared with) it. This
   branch doesn't block Phase 3, but every tick minted without a climb identity is one more
   row a future resolver has to backfill.
9. **`personalGrade` needs to be on the axis too.** "Your Rating" is stored as a raw
   unmapped string. Phase 7's sandbag index ("this area runs stiff for climbers under
   5'6″") and stiffness signals need personal grades comparable to consensus grades — i.e.
   a second mapping/FK. Cheap to add now, a migration later.
10. **The axis is difficulty-only.** Taste matching (goal 2) and suitability (goal 3) also
    need *style* (crimpy/slopey/reachy, angle, morpho lean) and *perceived* difficulty, not
    just consensus difficulty. That's ROADMAP Phase 5 (style-tag vocabulary + climber
    profiles). Worth noting so the grade axis isn't mistaken for "the recommendation
    substrate" — it's one dimension of it.

### Test & CI debt this branch highlights

11. **CI still never runs backend tests** — `playwright.yml` only does
    `spring-boot:run`. The new `TickOwnershipIntegrationTest` and any grade-mapping tests
    don't gate merges. Add `mvnw test` to CI.
12. **Guard the CSV ↔ enum ↔ generator contract** — a test that the CSV parses with the
    seeder's exact commons-csv settings, every `gradeSystem`/`discipline` value maps to a
    Java enum constant, and (ideally) regenerating produces the committed file. Today a
    generator change that adds an unknown enum value is only caught at boot, in production.
    (Partially covered now: `GradeSeedCoverageIntegrationTest` boots the real seeder over
    the committed CSV inside the test suite, so an unknown enum value or malformed row
    fails tests at context startup. The regenerate-reproduces-the-committed-file check
    is still open.)
13. **Test data couples to tuned values** — `TickOwnershipIntegrationTest` asserts
    `difficultyScore == 55.0`, a specific CSV opinion value, so a legitimate retune breaks
    the ownership suite. Assert *presence and monotonicity* of the score, not the exact
    number, in tests that aren't about the axis itself.
